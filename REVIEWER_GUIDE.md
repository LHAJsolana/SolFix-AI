# SolFix AI Reviewer Guide

## Product Explanation

SolFix AI is a Solana transaction diagnostics tool. A user pastes a transaction signature, chooses mainnet-beta or devnet, and SolFix retrieves RPC metadata, maps instructions and logs, isolates the failing instruction when evidence permits, classifies the error with deterministic rules, and produces a shareable report with repair guidance.

## Why Solana Is Essential

The product depends on Solana transaction signatures, Solana RPC responses, instruction and inner-instruction metadata, program logs, compute-unit data, wallet signing, and the devnet Memo Program for optional report-hash attestation.

## Deterministic Evidence vs AI Explanation

SolFix follows:

```text
RPC facts -> deterministic rules -> program intelligence -> AI explanation
```

AI can only rewrite explanatory text from structured evidence. It cannot change the transaction signature, network, error category, instruction index, program ID, confidence score, verification status, or evidence references.

## Six-Step Test Flow

1. Open `<DEPLOYED_APP_URL>`.
2. Click `Load example`.
3. Confirm the report shows a failed transaction diagnosis.
4. Switch between User mode and Developer mode.
5. Inspect evidence, instruction tree, and raw logs.
6. Open the shareable report URL.

## Real Transaction Flow

1. Open `/analyze`.
2. Paste a real failed devnet or mainnet-beta transaction signature.
3. Click `Inspect transaction`.
4. Confirm RPC retrieval, instruction normalization, evidence, confidence reasons, and report persistence.
5. Reopen the report URL after deployment restart when PostgreSQL is configured.

## Demo Transaction Flow

1. Open `/demo`.
2. Select `Compute budget exceeded`.
3. Confirm the deterministic fixture is labeled as demo data.
4. Inspect failure location, evidence rows, logs, and repair guidance.

## Memo Attestation Flow

1. Open a saved report.
2. Connect a devnet-funded browser wallet.
3. Click `Anchor Memo`.
4. Review the wallet transaction.
5. Reject once to verify graceful handling.
6. Retry, sign, send, and confirm.
7. Open the devnet Explorer link and confirm the Memo payload hash.

## Feature Matrix

| Feature | Status | How to test | External dependency | Known limitation |
| --- | --- | --- | --- | --- |
| Demo diagnosis | Complete | `/demo`, load any scenario | None | Demo signatures are fixtures, not chain transactions |
| Live RPC analysis | Complete | `/analyze` with valid signature | Solana RPC | Old/pruned transactions depend on RPC provider |
| Successful transaction handling | Complete | Analyze successful tx | Solana RPC | No failure diagnosis is generated by design |
| Deterministic classifiers | Complete | Unit tests and demo scenarios | None | Unknown custom errors remain probable/manual |
| Shareable reports | Complete | Open `/report/<slug>` | PostgreSQL for production durability | Local fallback writes `.data/solfix-reports.json` |
| Prisma persistence | Complete with configured DB | Set `DATABASE_URL`, migrate, restart, reopen report | PostgreSQL | Requires external DB verification |
| AI explanations | Optional | Set provider env vars | OpenAI or Anthropic | Deterministic output remains source of truth |
| Simulation API | Complete for bounded serialized tx | `POST /api/simulate` | Solana RPC | Does not accept arbitrary code or unsigned repair plans |
| Memo attestation | Complete for browser wallet flow | Report page, devnet wallet | Browser wallet and devnet SOL | Attests report hash only, not correctness |
| Health endpoint | Complete | `/api/health` | RPC/network | Health is a point-in-time check |

## Known Limitations

- Public RPC endpoints can rate-limit, prune history, or return partial metadata.
- Production/serverless durability requires `DATABASE_URL` and migrations; local development uses `.data/solfix-reports.json`.
- Browser wallet attestation requires a compatible wallet and devnet SOL.
- AI provider output is optional and evidence-constrained; it is not blockchain verification.
- Simulation success does not guarantee final mainnet landing.
